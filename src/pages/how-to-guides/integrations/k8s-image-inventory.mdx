# Kubernetes Image Inventory DevGuard Integration

[devguard-k8s-image-inventory](https://github.com/l3montree-dev/devguard-k8s-image-inventory) is a daemon that runs inside your Kubernetes cluster, watches all pods, generates a CycloneDX SBOM for each container image using [Trivy](https://github.com/aquasecurity/trivy), and reports it to DevGuard. Images are organised in DevGuard by namespace and workload — no manual asset registration required. When images disappear from the cluster they are automatically removed from DevGuard.

---

## How it works

The daemon watches all pods in the cluster. Whenever it discovers a container image it hasn't seen before, it:

1. Pulls the image and generates a CycloneDX SBOM with [Trivy](https://github.com/aquasecurity/trivy)
2. Reports the image to DevGuard via the [External Entity API](/reference/api/external-entity) (`verb=update`) — DevGuard creates or updates the project hierarchy and triggers a vulnerability scan
3. Annotates the pod so the same image is not re-scanned on the next run (unless `SBOM_IGNORE_ANNOTATIONS=true`)

When a pod with a particular image stops running and no other pod references the same image, the daemon sends a `verb=delete` to DevGuard. DevGuard cascades the deletion upward: artifact → asset version → asset → project, stopping at the first level that still has other data referencing it.

### How images are mapped in DevGuard

The daemon uses your Kubernetes namespace and workload names as external entity IDs. DevGuard organises them under the root project you configure:

```
Root project  (the DevGuard project you point DEVGUARD_PROJECT_URL at)
└── Namespace              → Sub-project  (e.g. "production")
    └── Workload name      → Asset        (e.g. "nginx")
        └── Image tag      → Asset version (e.g. "1.27")
            └── Image ref  → Artifact     (e.g. "nginx:1.27")
```

This means after installation you will see your Kubernetes workloads appear automatically in DevGuard, grouped by namespace, each with their running images and associated vulnerability findings.

### Operating modes

Two modes are available:

- **Real-time** (default): a pod informer triggers analysis immediately when pods change, plus a full sync on startup. Good for most clusters.
- **Cron**: scans all pods on a schedule. Useful if you want to batch scans during off-peak hours. Configure via `SBOM_CRON`.

## Requirements

- Kubernetes 1.29+
- A DevGuard account and an existing root project
- A DevGuard API token — see [Personal Access Tokens](/getting-started/use-devguard-api)

## Installation

### 1. Create the namespace and token secret

```bash
kubectl create namespace devguard

kubectl create secret generic devguard-k8s-image-inventory \
  --namespace devguard \
  --from-literal=devguard-token=<your-devguard-token>
```

### 2. Apply the RBAC manifests

```bash
kubectl apply -f deploy/rbac.yaml
```

This creates a `ServiceAccount`, `ClusterRole`, and `ClusterRoleBinding` with the minimum read access required (pods, namespaces, secrets, ReplicaSets).

### 3. Configure and apply the deployment

Edit `deploy/deployment.yaml` and set `DEVGUARD_PROJECT_URL` to your DevGuard project:

```yaml
env:
  - name: DEVGUARD_PROJECT_URL
    value: "https://api.devguard.org/api/v1/organizations/<org-slug>/projects/<project-slug>/external"
```

The daemon appends its own provider ID to this URL. You can choose any provider ID — it namespaces all entries this daemon creates in DevGuard.

Then apply:

```bash
kubectl apply -f deploy/deployment.yaml
```

<Callout type="info">
  For self-hosted DevGuard instances replace `https://api.devguard.org` with your instance's API base URL.
</Callout>

## Configuration

All options are read from environment variables (or a YAML config file).

| Environment variable | Default | Description |
|----------------------|---------|-------------|
| `DEVGUARD_PROJECT_URL` | — | DevGuard project URL (required) |
| `DEVGUARD_TOKEN_FILE` | — | Path to a file containing the API token. The deployment mounts the secret at `/etc/devguard/token`. |
| `SBOM_CRON` | `""` | Cron expression for scheduled mode. When empty, real-time mode is used. |
| `SBOM_POD_LABEL_SELECTOR` | `""` | Kubernetes label selector to restrict which pods are scanned |
| `SBOM_NAMESPACE_LABEL_SELECTOR` | `""` | Kubernetes label selector to restrict which namespaces are scanned |
| `SBOM_IGNORE_ANNOTATIONS` | `false` | When `true`, re-scans images even if already processed |
| `SBOM_FALLBACK_PULL_SECRET` | `""` | Name of a pull secret (in the daemon namespace) used as fallback for private registries |
| `SBOM_REGISTRY_PROXY` | `[]` | Registry proxy mappings, e.g. `docker.io=my-mirror.example.com`. Repeatable. |
| `SBOM_JOB_TIMEOUT` | `3600` | Maximum seconds a scan job may run |
| `SBOM_VERBOSITY` | `info` | Log level: `debug`, `info`, `warn`, `error` |

## Security

- Runs as non-root (`UID 53111`) on a read-only root filesystem
- The DevGuard token is mounted from a Kubernetes secret — never passed as an environment variable
- All Linux capabilities are dropped; privilege escalation is disabled
- Seccomp profile set to `RuntimeDefault`
